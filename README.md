# 테스트 전략 및 가이드라인

이 문서는 이 프로젝트에서 어떤 기준으로 테스트를 작성하고 유지할지에 대한 전략을 정리한 것입니다.
주요 참고 문서는 "Sustainable Rails"이며, 해당 문서에서 제안하는 서비스 객체 중심의 비즈니스 로직 구조와 테스트 전략을 기반으로 하고 있습니다.

## 비즈니스 로직 구조에 대한 기본 철학

Rails는 컨벤션 중심의 프레임워크로, 컨트롤러나 모델(특히 ActiveRecord 모델) 내부에 로직을 두는 방식도 충분히 의미가 있습니다. 실제로 이 프로젝트 초기에는 컨트롤러나 모델 내부에 비즈니스 로직을 직접 작성하여 사용하는 방식으로 개발을 진행했었습니다.

그런데 프로젝트를 팀원들과 함께 운영하면서 기능 변경이나 요구사항이 계속 바뀌는 상황을 겪게 되었고, 그 과정에서 모델 내부에 포함되어 있던 로직들이 여러 곳에서 재사용되다 보니 작은 수정을 하더라도 어디에 영향을 미칠지 파악하기 어려웠던 경험이 있었습니다. 특히 ActiveRecord 모델은 데이터 접근과 상태를 함께 다루다 보니, 순수한 도메인 로직과 섞이게 되는 일이 자주 있었습니다.

또한, 루비는 타입스크립트와는 달리 정적 타입 시스템이 없기 때문에 문법적으로 틀리거나, 코드가 잘못 수정되어도 바로 에러가 드러나지 않는 경우가 있었고, 이런 부분도 테스트가 없다면 파악하기 어려운 지점이었습니다. 이런 경험을 하면서 테스트 코드의 필요성도 자연스럽게 체감하게 되었고, 결과적으로 서비스 객체로 로직을 분리하고, 그 단위에서 테스트를 작성하는 방식이 현재 상황에 더 적합하다는 판단을 하게 되었습니다.

이러한 이유들로 인해 이 프로젝트에서는 다음과 같은 방식을 채택했습니다:

- 비즈니스 로직은 컨트롤러나 모델이 아닌 **서비스 객체(Plain Old Ruby Object)** 에서 처리합니다.
- 컨트롤러는 **입력 받은 파라미터를 서비스 객체에 전달하고 결과를 렌더링**하는 역할만 수행합니다.
- 예외 대신 `success?`, `failure?`, `data`, `error_message` 등을 갖는 **결과 객체 패턴**을 사용하여 흐름을 명확하게 분기합니다.

예시:

```ruby
result = Auth::LoginProcessor.new(params).call

if result.success?
  result.data
else
  result.error_message
end
```

이런 구조는 테스트 코드와 함께 사용할 때 특히 안정감을 느낄 수 있었고, 나중에 유지보수하거나 확장할 때에도 상대적으로 구조가 단순하게 유지될 수 있는 장점이 있었습니다.

## 테스트 구성 기준

| 구분       | 목적             | 위치                  | 작성 여부             |
| -------- | -------------- | ------------------- | ----------------- |
| 유닛 테스트   | 서비스 객체 단위 테스트  | `test/use_cases/`   | 작성합니다 ✅           |
| 통합 테스트   | 사용자 흐름 단위 테스트  | `test/integration/` | 작성합니다 ✅ (해피패스 중심) |
| 컨트롤러 테스트 | 파라미터/응답 형식 테스트 | 작성하지 않습니다 ❌         | 이유: 서비스 테스트로 대체함  |

## 통합 테스트 작성 기준

- 통합 테스트는 실제 API 흐름이 잘 연결되는지만 확인하면 충분합니다.
- 파라미터 오류나 유효성 에러 등은 모두 서비스 테스트에서 다루므로, 중복 테스트는 작성하지 않습니다.
- 통합 테스트는 **해피패스 위주**로 작성하며, 실패 흐름은 생략합니다.

### 예시 흐름

`test/integration/api/v1/auth_flow_test.rb`

- 회원가입 → OTP 발송 → OTP 인증 → 로그인 → 토큰 재발급 → 로그아웃

## 디렉토리 구조 예시

```
test/
├── use_cases/
│   └── auth/
│       ├── login_processor_test.rb
│       ├── register_user_test.rb
│       └── otp_processor_test.rb
├── integration/
│   └── api/v1/
│       └── auth_flow_test.rb
```

## 기타

NestJS로 개발했을 때는 테스트 작성이 막연하게 느껴졌습니다. 컨트롤러 → 서비스 → 리포지토리 구조가 관례처럼 자동 생성되다 보니, 자연스럽게 테스트 파일 수도 많아졌고, 컨트롤러와 서비스 간 로직이 정말 잘 연결되는지, 혹은 서비스에 비즈니스 로직을 몰아넣는 것이 맞는지에 대해 프레임워크가 명확한 방향을 제시해주지 않아 개발자 간 논쟁이 발생하는 경우도 종종 있었습니다.

또한 NestJS에서는 실제 DB를 테스트하지 않고 비즈니스 로직만을 테스트하기 위해 인터페이스를 두거나 모킹 라이브러리를 사용하는 등의 방식이 일반적이다 보니, 테스트 환경을 구성하는 복잡도가 높았습니다. 아키텍쳐 문제에 대한 고민을 하는 것은 나름의 뿌듯함은 있었지만, 생산성과는 거리가 멀다는 느낌을 받았습니다.

그에 비해 Rails는 ActiveRecord 패턴을 기반으로 한 구조를 취하고 있고, 테스트도 DB와의 상호작용을 당연하게 받아들이는 관점이 있어서 테스트 작성이 오히려 단순하고 명확하게 느껴졌습니다. 실제로 이 프로젝트에서는 TEST DB를 대상으로 서비스 단위 테스트를 작성하고 있는데, 모킹 없이 로직 흐름 그대로 테스트할 수 있다는 점에서 최근 들어 가장 즐겁게 개발하고 있습니다.
